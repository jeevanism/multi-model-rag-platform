# Database Design (Postgres + pgvector)

This file documents the current database design for the project, with special focus on the vector/RAG schema and why some choices (like vector dimension = `8`) were made.

Use this as the reference before changing:
- embedding models
- vector dimensions
- migration strategy
- retrieval behavior

## 1. Database Stack

- Database: PostgreSQL
- Vector extension: `pgvector` (`CREATE EXTENSION IF NOT EXISTS vector;`)
- Local runtime: Docker Compose (`pgvector/pgvector:pg16`)
- Cloud runtime: Cloud SQL Postgres 16 (with `vector` extension enabled)

## 2. Core Schema (Current)

### RAG tables

- `documents`
  - stores ingested source document metadata + original content
- `chunks`
  - stores chunked text derived from `documents`
  - linked by `document_id`
- `embeddings`
  - stores vector embedding per chunk
  - linked by `chunk_id`
  - includes `provider`, `model`, and `embedding`

### Eval tables

- `eval_run`
  - summary for each eval execution
- `eval_run_case`
  - per-case result rows
  - includes judge score columns (correctness / groundedness / hallucination)

## 3. Current Vector Design (Important)

### Vector column shape

Current schema stores embeddings as:

```sql
VECTOR(8)
```

This is intentionally small and **not production embedding dimensionality**.

### Why `8` dimensions right now?

The project started with deterministic stub embeddings to prove the RAG pipeline mechanics:
- ingestion
- vector storage
- retrieval query
- grounding
- citations

Using `8` dimensions gave us:
- easy local development
- stable tests
- no large vectors in early iterations
- faster iteration while the LLM + embeddings providers were still stubbed

## 4. Retrieval Query Design (Current)

Retrieval uses pgvector distance ordering:

- operator: `<->` (distance)
- sorts ascending
- returns `top_k`

Current behavior:
- query text is embedded using the active embedding provider
- vector is cast into `vector(8)` in SQL
- nearest chunks are returned with a distance score

## 5. Embedding Providers (Current Behavior)

The code now supports embedding mode configuration with fallback:

- `EMBEDDING_PROVIDER_MODE=stub|auto|real`
- `EMBEDDING_PROVIDER=gemini|openai`

### Modes

- `stub`
  - deterministic test embeddings (default for safety)
- `auto`
  - uses real embeddings only if corresponding API key is present
  - falls back to stub otherwise
- `real`
  - requires API key and provider SDK
  - fails fast if missing

## 6. Important Constraint: Vector Dimension Must Match Schema

This is the key rule:

- The embedding vector length generated by code **must match** the DB column type dimension.

Right now:
- code uses `EMBEDDING_DIM = 8`
- DB column is `VECTOR(8)`

If you switch to a real embedding model with native dimensions like `768`, `1024`, `1536`, or `3072`, you cannot just flip an env var and continue using the same column definition.

## 7. Future Upgrade Plan (Production-Quality Embeddings)

When moving from the compatibility bridge (`8` dims) to production-size embeddings:

### Option A (recommended): New migration + re-index

1. Create a migration that changes embedding storage to the new dimension
   - example: `VECTOR(768)` or `VECTOR(1536)`
2. Rebuild/recreate embeddings for all chunks
3. Re-run RAG tests and evals

Notes:
- In many cases, easiest path is:
  - truncate/rebuild `embeddings` table
  - re-ingest documents
- Keep `documents` and `chunks` if possible

### Option B: New embeddings table (safer for migration experiments)

Create a parallel table (example: `embeddings_v2`) with:
- new vector dimension
- provider/model metadata

Then:
- backfill embeddings
- switch retrieval query to `embeddings_v2`
- remove old table later

This is safer if you want rollback during live demos.

## 8. Current Tradeoff (Documented)

The current `VECTOR(8)` design is a **deliberate compatibility bridge**:

What it proves:
- end-to-end RAG pipeline works
- pgvector integration works
- retrieval + citations + cloud deployment work

What it does not prove:
- semantic retrieval quality of production embeddings

This tradeoff is acceptable for iterative development and portfolio demonstration, but should be upgraded before claiming production-grade retrieval quality.

## 9. Operational Notes (Cloud)

Cloud RAG/evals require:
- Cloud SQL instance running
- `DATABASE_URL` in Secret Manager
- Cloud Run attached to Cloud SQL
- migrations applied (including RAG/evals schema)

If Cloud SQL is paused:
- `/health`, `/chat`, `/chat/stream` (without `rag=true`) can still work
- DB-backed endpoints fail:
  - `/ingest/text`
  - `/evals/*`
  - `/chat` with `rag=true`
